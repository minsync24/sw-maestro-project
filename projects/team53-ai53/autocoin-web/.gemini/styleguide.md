# Coin Agent Frontend Code Review Convention

이 문서는 Coin Agent Frontend 프로젝트의 코드 리뷰 기준을 정의한다.  
리뷰는 한국어로 작성하며, 문제점이 있을 경우 원인과 개선 방향을 함께 제시한다.

---

## 1. 프로젝트 기본 원칙

- Coin Agent Frontend는 React 19, TypeScript 기반으로 작성한다.
- 스타일링은 CSS Module 방식을 따른다.
- FE는 Binance Spot Testnet 전용 UI를 제공한다.
- FE는 모든 외부 API 호출을 반드시 BE를 통해 수행한다.
- FE는 Binance API Key 또는 Secret 원문을 입력받거나 화면에 표시하지 않는다.
- FE는 실거래 기능처럼 오해될 수 있는 UI와 문구를 사용하지 않는다.
- 모든 화면에는 Binance Spot Testnet 환경임을 명확히 드러내야 한다.
- 수익 보장, 투자 권유, 공격적 투자 표현을 사용하지 않는다.

---

## 2. React 컴포넌트 설계 규칙

- 컴포넌트는 하나의 명확한 책임을 가져야 한다.
- 명세에 정의된 주요 컴포넌트의 목적과 표시 내용을 유지한다.
- 컴포넌트 내부에 API 호출, 데이터 정규화, 복잡한 상태 판단 로직을 과도하게 포함하지 않는다.
- 폼, 카드, 패널, 차트, 로그 영역의 책임을 분리한다.
- 주문 생성, 주문 상태 조회, 주문 취소 UI는 서로 독립적으로 관리한다.
- WebSocket 시세 정보는 보조 기능으로만 표시하며, 주요 시세 조회 흐름과 혼동되지 않게 한다.
- 컴포넌트 이름은 역할을 명확히 드러내야 한다.
- 공통 UI 컴포넌트와 도메인 컴포넌트를 구분한다.
- props가 과도하게 많아지는 경우 컴포넌트 책임 분리 또는 props 객체화를 검토한다.
- 조건부 렌더링은 사용자에게 보이는 상태가 명확하도록 작성한다.
- 리스트 렌더링 시 index를 key로 사용하는 것을 지양하고, 안정적인 식별자를 사용한다.

---

## 3. React 상태 관리 규칙

- 서버/API 상태와 클라이언트 UI 상태를 구분한다.
- 주문 폼 입력 상태, 조회 결과 상태, WebSocket 상태, 로그 상태를 무분별하게 하나의 상태 객체에 섞지 않는다.
- 파생 가능한 값은 중복 저장하지 않는다.
- 상태 전이는 예측 가능해야 한다.
- API 실패, 주문 실패, WebSocket 연결 실패 시 이전 성공 상태가 현재 상태처럼 오해되지 않도록 관리한다.
- 마지막 정상 데이터를 표시하는 경우 최신 데이터가 아님을 함께 표시한다.
- 사용자가 새로운 symbol을 선택하면 이전 symbol 기준 데이터가 stale 상태로 남지 않도록 한다.
- stream 상태와 REST 조회 상태를 분리해 관리한다.

---

## 4. React Effect 및 비동기 처리 규칙

- `useEffect`는 외부 시스템과의 동기화가 필요한 경우에만 사용한다.
- 단순 계산이나 파생값 생성을 위해 불필요하게 `useEffect`를 사용하지 않는다.
- API 호출, WebSocket 연결, timer, event listener는 cleanup을 반드시 고려한다.
- 동일 API 요청이 중복 실행되지 않도록 방지한다.
- polling은 기본 동작으로 사용하지 않으며, 명세에 따라 수동 조회 버튼 기반 흐름을 우선한다.
- WebSocket 연결은 컴포넌트 unmount 또는 symbol 변경 시 정리되어야 한다.
- 비동기 요청 실패 시 오류 상태를 사용자에게 표시해야 한다.
- loading 상태가 끝나지 않거나, 실패 후에도 성공 UI가 유지되는 상태를 만들지 않는다.

---

## 5. TypeScript 규칙

- 핵심 데이터 구조는 명시적인 타입으로 정의한다.
- `any` 사용을 지양한다.
- API 응답 타입과 화면 표시용 타입을 구분한다.
- 주문 상태, 주문 방향, 주문 타입, UI 상태처럼 제한된 값은 union type으로 관리한다.
- nullable 값은 명시적으로 처리한다.
- 타입 단언은 최소화한다.
- 문자열 상태값을 여러 파일에 중복 정의하지 않는다.
- Binance 응답의 숫자 문자열은 화면 표시 전 정규화하되, 원본 값을 확인할 수 있는 구조를 유지한다.

---

## 6. API 호출 규칙

- FE는 Binance API를 직접 호출하지 않는다.
- 모든 API 호출은 명세에 정의된 BE 엔드포인트를 통해 수행한다.
- API 호출 로직은 UI 컴포넌트와 분리한다.
- API 요청에는 로딩, 성공, 빈 상태, 부분 오류, 전체 오류 상태를 구분해야 한다.
- API 실패 시 사용자가 이해할 수 있는 오류 메시지를 표시한다.
- Binance 에러 코드는 가능한 경우 사용자에게 함께 표시한다.
- 핵심 API 실패 시 실거래가 아님을 다시 명확히 안내하고 재시도 가능 상태를 제공한다.
- 마지막 정상 데이터를 보여줄 경우 최신 데이터가 아님을 표시한다.
- 자동 갱신은 기본 동작으로 사용하지 않고, 수동 조회 버튼 기반 흐름을 우선한다.

---

## 7. 주문 테스트 규칙

- 주문 버튼은 필수 파라미터가 모두 채워진 경우에만 활성화한다.
- 시장가 매수, 시장가 매도, 지정가 주문의 필수 입력값을 구분해 검증한다.
- 주문 타입에 맞지 않는 파라미터 조합은 요청 전에 차단한다.
- 주문 결과는 즉시 로그 영역에 표시한다.
- 주문 상태는 명세에 정의된 상태 배지를 통해 표시한다.
- 주문 취소는 symbol과 orderId 또는 origClientOrderId가 유효할 때만 요청 가능해야 한다.
- 주문 테스트 UI는 실제 거래가 아닌 Testnet 주문임을 명확히 표시해야 한다.
- 주문 실패 시 실패 원인과 Binance 에러 코드를 함께 보여준다.

---

## 8. WebSocket 및 시세 표시 규칙

- WebSocket 데이터는 보조 정보로만 사용한다.
- 실시간 데이터는 StreamStatusCard에서만 명확히 구분해 표시한다.
- REST 기반 시세와 WebSocket 기반 이벤트를 혼동하지 않게 표현한다.
- stream 이름은 소문자로, REST symbol은 대문자로 다룬다.
- WebSocket 연결 상태, stream name, latest event를 표시해야 한다.
- WebSocket 연결 실패 시 전체 화면 오류로 처리하지 않는다.
- WebSocket 카드의 오류는 부분 오류로 처리한다.
- stream 수신 중, 연결 중, 재연결 중, 연결 끊김 상태를 구분해 표시한다.

---

## 9. UI 상태 처리 규칙

- 모든 주요 데이터 영역은 로딩, 빈 상태, 성공, 부분 오류, 전체 오류를 고려해야 한다.
- 로딩 상태에서는 스켈레톤 또는 spinner를 표시한다.
- 빈 상태에서는 사용자가 다음 행동을 이해할 수 있는 안내를 제공한다.
- 성공 상태에서는 카드, 차트, 표 등 명세에 맞는 UI를 표시한다.
- 부분 오류에서는 오류 배너와 마지막 정상 데이터를 함께 표시할 수 있다.
- 전체 오류에서는 재시도 안내와 Testnet 환경 경고를 함께 표시한다.
- 오류 상태를 숨기거나 성공 상태처럼 표시하지 않는다.
- 같은 화면 안에서 여러 요청이 동시에 발생할 수 있으므로, 요청별 상태를 구분해 표시한다.

---

## 10. AI-oriented UI/UX 규칙

- AI Agent의 작업 상태는 사용자에게 명확히 전달해야 한다.
- AI 응답 생성, 분석, 요약, 로그 해석, 리포트 생성 중인 상태를 구분해 표시한다.
- streaming 응답이 진행 중이면 답변이 완성되지 않았음을 명확히 표시한다.
- streaming 중인 텍스트와 완료된 텍스트를 시각적으로 구분할 수 있어야 한다.
- AI 응답 생성 중에도 사용자가 현재 작업 상태를 이해할 수 있어야 한다.
- AI 응답 실패 시 실패 상태와 재시도 가능 여부를 표시한다.
- AI 응답이 주문 실행 여부를 결정하는 것처럼 보이면 안 된다.
- AI가 생성한 설명은 실행 결과, Binance 응답, 로그 데이터와 구분되어야 한다.
- AI 설명은 원본 데이터나 로그를 대체하지 않고 보조 정보로 표시해야 한다.
- AI가 불확실한 내용을 확정적으로 표현하지 않도록 UI 문구를 제한한다.
- AI 응답이 비어 있거나 지연될 경우 fallback 메시지를 제공한다.
- 사용자가 AI 생성 결과와 시스템 원본 결과를 혼동하지 않도록 라벨을 분리한다.
- AI 작업 중 사용자가 취소하거나 다른 작업으로 이동할 수 있는 흐름을 고려한다.
- AI Agent가 여러 단계를 수행하는 경우 현재 단계를 표시한다.
- Agent 상태는 최소한 대기 중, 요청 처리 중, 응답 생성 중, 완료, 실패 상태를 구분한다.

---

## 11. UI/UX 문구 규칙

- 항상 Binance Spot Testnet 환경임을 상단 또는 주요 영역에 표시한다.
- 실거래가 아님을 경고 배너로 명확히 표시한다.
- 수익 보장, 확정적 예측, 투자 권유 문구를 사용하지 않는다.
- 버튼 문구는 사용자가 수행하는 동작을 명확히 드러내야 한다.
- 주문 관련 문구는 실제 거래와 Testnet 주문을 혼동시키지 않아야 한다.
- 에러 메시지는 기술적 원인과 사용자 행동 안내를 함께 제공한다.
- 숫자 데이터에는 단위를 명확히 표시한다.
- 잔고, 가격, 수량, 주문 금액은 사용자가 오해하지 않도록 포맷팅한다.

---

## 12. 스타일링 규칙

- 스타일링은 CSS Module 방식을 따른다.
- 컴포넌트별 스타일은 `*.module.css` 파일로 분리한다.
- CSS Module import 이름은 `styles`로 통일한다.
- 전역 CSS 사용은 reset, font, CSS variable, body 스타일에 한정한다.
- 인라인 스타일 사용을 지양한다.
- 반복되는 색상, 간격, radius, typography 값은 CSS variable로 관리할 수 있다.
- className은 시각적 형태보다 역할과 의미를 기준으로 작성한다.
- 위험, 경고, 성공, 비활성 상태의 스타일은 일관되게 관리한다.
- 위험 상태를 색상만으로 표현하지 않는다.
- 주문 상태 배지는 상태별로 일관된 스타일을 사용한다.

---

## 13. 보안 및 민감 정보 규칙

- API Key와 Secret은 클라이언트 코드에 포함하지 않는다.
- FE는 API Key 원문을 입력받거나 재표시하지 않는다.
- 환경 변수 상태는 설정 여부만 표시한다.
- 민감 정보가 로그, 에러 메시지, 화면에 노출되지 않도록 한다.
- BE에서 전달받은 환경 상태도 사용자에게 필요한 최소 정보만 표시한다.
- 실거래 URL과 Testnet URL이 혼동되지 않도록 base URL 표시를 명확히 한다.

---

## 14. 코드 리뷰 우선순위

코드 리뷰는 다음 순서로 우선 검토한다.

1. FE가 외부 Binance API를 직접 호출하지 않는가
2. API Key 또는 Secret이 클라이언트에 노출되지 않는가
3. Testnet 환경임이 명확히 표시되는가
4. 주문 테스트 파라미터 검증이 올바른가
5. 주문 결과와 오류가 명확히 표시되는가
6. UI 상태가 로딩, 빈 상태, 성공, 부분 오류, 전체 오류로 구분되는가
7. AI Agent의 작업 상태와 streaming 상태가 명확히 표시되는가
8. TypeScript 타입이 명확하게 정의되어 있는가
9. React 컴포넌트 책임과 상태 관리가 적절히 분리되어 있는가
10. CSS Module 규칙이 지켜지는가

---

## 15. Blocking Review 대상

다음 항목은 반드시 수정 요청한다.

- FE에서 Binance API를 직접 호출하는 코드
- API Key 또는 Secret이 클라이언트 코드에 포함된 코드
- API Key 원문을 입력받거나 표시하는 UI
- 실거래로 오해될 수 있는 문구 또는 화면
- 주문 필수 파라미터 검증 없이 주문 요청을 보내는 코드
- 주문 실패를 성공처럼 표시하는 코드
- Binance 에러 코드를 숨기는 코드
- AI 응답이 주문 실행 여부를 결정하는 것처럼 보이는 UI
- streaming 중인지 완료된 답변인지 구분되지 않는 AI 응답 UI
- `any`로 핵심 API 응답이나 주문 타입을 처리하는 코드
- CSS Module을 사용하지 않고 전역 스타일을 남발하는 코드
