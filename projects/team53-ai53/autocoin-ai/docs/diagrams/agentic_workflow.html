이 페이지는 plan/MASTER_PLAN.md, plan/CONTRACTS.md, plan/PHASES.md 기준으로 새 AI 흐름을 한 번에 이해하기 위한 문서다. 핵심은 주문 실행 자동화가 아니라 자연어 이해, 원칙 기반 판단, 안전 검증, 설명 가능한 trace다.

사용자 입력폼 입력뿐 아니라 “BTC 5만원 사줘” 같은 문장도 받는다.
AI 판단트레이더 원칙 RAG를 참고해 매매 제안을 만든다.
코드 검증Risk Gate가 한도, 잔고, 변동성을 결정론적으로 막는다.

1. Before / After

기존은 이미 정리된 주문 dict를 받아 규칙 검증만 했다. 새 흐름은 앞에 AI 이해 단계와 원칙 기반 strategy가 붙는다.

기존 흐름

사용자가 주문 필드를 이미 채워줬다고 가정한다. AI 느낌보다는 계약 검증에 가깝다.

policy → risk → evaluator

Agentic MVP 흐름

자연어를 이해하고, 트레이더 원칙을 찾아보고, LLM이 제안한 뒤, 코드가 마지막 안전 검증을 한다.

intake → policy + RAG → strategy → risk_gate → evaluator

2. 한 요청이 지나가는 길

예시는 “리버모어 스타일로 BTCUSDT 100 USDT 매수” 또는 “워뇨띠처럼 BTC 5만원 사줘” 같은 입력이다. 모든 단계는 decision_trace와 verification_checks에 흔적을 남긴다.

intake말을 주문 데이터로 바꿈
사용자 문장을 해석한다.

text나 raw_text가 있으면 Gemini로 파싱한다. 기존 symbol/side/type dict도 그대로 지원한다.

출력: normalized_order_intent, trader_id, inferred_persona
실패: 모호하면 HOLD_INPUT_AMBIGUOUS, schema가 깨지면 FAILED
policy + RAG정책과 원칙을 붙임
이 요청을 어떤 기준으로 판단할지 준비한다.

persona 한도와 허용 심볼을 붙이고, knowledge/{trader_id}/principles.md에서 관련 원칙을 가져온다.

출력: persona_bounds, trader_principles
현재: 키워드 RAG, 후속: embedding vector retrieval
strategyLLM 매매 제안
트레이더 원칙을 근거로 BUY/HOLD를 제안한다.

Gemini 2.5 Pro가 원칙, persona, 주문 의도를 보고 conviction과 이유를 만든다.

출력: llm_proposal
핵심 필드: action, size_usd, conviction, matched_principle_titles
risk_gate결정론 안전장치
LLM 제안을 그대로 믿지 않고 코드로 막는다.

확신도, 주문 한도, 허용 심볼, 잔고, 변동성을 순서대로 검사한다. 여기서는 LLM을 부르지 않는다.

출력: risk_assessment, risk_tool_calls, lifecycle_status
통과하면 READY_FOR_BE, 애매하면 HOLD, 불가하면 NO_ORDER
evaluator최종 설명 정리
판단을 뒤집지 않고 사용자가 이해할 말로 바꾼다.

risk_gate가 정한 결과를 echo하고, 왜 그런지 2~3문장으로 설명한다. LLM 실패 시 코드 요약으로 대체한다.

출력: evaluator_review.summary, user_message
모든 종료 케이스가 evaluator를 거쳐 설명을 만든다.
BE revalidation실행 전 최종 검증
AI가 통과시켜도 BE가 다시 확인한다.

READY_FOR_BE는 체결 성공이 아니라 BE에게 넘길 준비가 됐다는 뜻이다. 실제 제출 전 BE가 다시 검증한다.

성공: execution_result
차단: BE_REJECTED + be_rejection_evidence

3. 데모에서 보일 시나리오

같은 파이프라인을 지나지만, 입력과 원칙, risk_gate 결과에 따라 최종 상태가 달라진다.

READY_FOR_BE

워뇨띠 buy

“BTC 5만원치 사줘”를 자연어로 받아 BTCUSDT 시장가 매수 제안으로 바꾼다.

  • 워뇨띠 원칙 중 리스크 관리, 추세 확인 매칭
  • conviction이 persona 기준 이상
  • 잔고와 변동성 검사 통과
HOLD

낮은 확신도

횡보장이나 근거 부족이면 strategy가 낮은 conviction을 내고 risk_gate가 보류한다.

  • conviction < min_conviction
  • HOLD_LOW_CONVICTION
  • evaluator가 “왜 관망인지” 설명
NO_ORDER

한도 초과

“보수적으로 BTC 200만원”처럼 persona 한도를 넘으면 주문을 만들지 않는다.

  • CONSERVATIVE max_order_usd 초과
  • risk_gate에서 즉시 차단
  • BE까지 가지 않는다

4. 말이 어려운 것들, 가볍게 풀면

문서에 자주 나오는 단어를 구현 관점으로 짧게 풀었다.

intake

사람 말을 주문 데이터로 바꾸는 입구다. text/raw_text가 있으면 AI가 해석하고, 기존 dict는 그대로 통과시킨다.

RAG

LLM이 그냥 상상하지 않도록 원칙 문서를 먼저 찾아 넣는 방식이다. 워뇨띠와 리버모어 원칙이 여기 들어간다.

strategy

AI가 “살지, 말지, 얼마나 확신하는지”를 제안하는 단계다. 최종 실행 권한은 없다.

risk_gate

LLM 제안을 코드로 검사하는 문이다. 확신도, 잔고, 한도, 변동성을 결정론적으로 본다.

evaluator

심판이 아니라 설명 정리자다. 결과를 바꾸지 않고 사용자가 이해할 문장과 trace 요약을 만든다.

LangSmith trace

각 노드와 LLM 호출이 어떻게 이어졌는지 보는 실행 기록 화면이다. 데모에서는 “AI가 뭘 했는지” 보여주기 좋다.

5. 최종 상태를 어떻게 읽으면 되나

상태값은 사용자에게 그대로 보여주기보다 evaluator 설명과 함께 보여주는 게 좋다.

READY_FOR_BE

AI와 risk_gate 기준으로는 넘길 수 있다. 아직 체결 성공이 아니다. BE 재검증이 남아 있다.

HOLD

지금은 보류다. 모호한 입력, 낮은 확신도, 변동성, 데이터 부족 등이 이유가 될 수 있다.

NO_ORDER

주문을 만들지 않는다. 허용 심볼이 아니거나 persona 한도를 넘는 경우가 대표적이다.

FAILED

입력 계약, LLM schema, 필수 필드 같은 기술적 문제가 깨진 상태다.

BE_REJECTED

AI는 넘겼지만 BE의 최종 검증에서 막힌 상태다. 실제 실행 전 마지막 안전망이다.

REPORT_READY

실행 또는 차단 결과까지 사용자에게 설명할 준비가 끝난 상태다.

핵심: PASS는 내부 stage 통과 표시일 뿐 사용자에게 “주문 성공”처럼 보여주면 안 된다.

6. 구현되면 주로 바뀌는 코드

전체를 외울 필요는 없다. 아래 파일들이 새 workflow의 뼈대다.

src/autocoin_ai/graph.py노드 순서를 intake → policy → strategy → risk_gate → evaluator로 연결한다.
src/autocoin_ai/models.pytrader_id, llm_proposal, risk_assessment, evaluator_review 같은 AI 상태 필드를 추가한다.
src/autocoin_ai/nodes/intake.py자연어 입력과 기존 dict 입력을 모두 처리한다.
src/autocoin_ai/nodes/strategy.pyRAG 원칙을 넣어 Gemini 2.5 Pro가 매매 제안을 만들게 한다.
src/autocoin_ai/nodes/risk_gate.pyLLM 없이 코드로 안전 조건을 검사한다.
src/autocoin_ai/nodes/evaluator.py최종 사용자 메시지와 run summary를 만든다.
src/autocoin_ai/rag/retriever.py워뇨띠/리버모어 원칙 문서에서 관련 chunk를 찾아온다.
knowledge/*/principles.md트레이더별 판단 원칙 데이터셋이다. 현재 wonyotti, livermore가 있다.
