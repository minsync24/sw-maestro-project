1 3-Tier 경계 — 누가 뭘 하나

이 시스템은 FE / BE / AI 셋이 분명히 분리되어 있어요. autocoin-ai 저장소는 그중 AI 한 칸만 책임지지만, 이 칸이 BE·FE와 어떻게 맞물리는지 알아야 코드가 이해돼요.

🖥 FE

사용자 입력 + 상태 표시
  • 주문 테스트 요청 폼
  • run_id·hold_reason·decision_trace 표시
  • resume용 보완 입력 재전송
안 함: Binance 직접 호출 / API 키 다루기 / 서명 / 실행 승인

⚙️ BE

정책 retrieval + Binance 호출 + 최종 실행
  • policy_context 만들어 AI에 넘김
  • AI proposal을 받아 deterministic 재검증
  • Binance Testnet에 서명·제출
  • execution_result / be_rejection_evidence 회신
안 함: AI 판단을 무검증으로 신뢰 / production host 사용

2 AI 내부 — LangGraph 4 노드

AI가 "추론"한다고 부르는 부분의 정체. 각 노드를 클릭하면 룰을 풀어드릴게요. LLM 호출 없이 결정론적 if/else로 동작합니다.

1️⃣ policy

nodes/policy.py
요청 검증·정규화 + 정책 grounding
normalized_order_intent policy_context

2️⃣ risk

nodes/risk.py
잔고·시장·심볼·금액 4단계 게이트
HOLD / NO_ORDER / READY_FOR_BE

3️⃣ evaluator

nodes/evaluator.py
근거(verification_checks) 충분성 재평가
READY_FOR_BE 유지 HOLD 강등

4️⃣ execution

nodes/execution.py
BE 결과 → 사용자 리포트 (별도 그래프)
REPORT_READY BE_REJECTED
policy_node — 진입 노드
request_context 필수 필드 누락 → INITIAL_REQUEST_CONTRACT_FAILED FAILED
symbol·side·type 대문자 정규화 + policy_refs 기록 → ORDER_INTENT_NORMALIZED PASS → risk
risk_node — 4단계 게이트 (위에서 아래로 순서대로)
market_snapshot_fresh == false → STALE_MARKET_SNAPSHOT HOLD_DATA_INSUFFICIENT
requires_review == true → HUMAN_REVIEW_REQUIRED HOLD_REVIEW_REQUIRED
symbol ∉ {BTCUSDT, ETHUSDT} → SYMBOL_NOT_ALLOWED NO_ORDER
quantity / quoteOrderQty <= 0 → ORDER_AMOUNT_INVALID NO_ORDER
모든 체크 통과 → ALL_CHECKS_PASSED READY_FOR_BE
evaluator_node — 마지막 검열
verification_checks에 policy=pass + risk=pass 둘 다 있어야 함. 하나라도 누락 → EVIDENCE_INSUFFICIENT HOLD_DATA_INSUFFICIENT
근거 둘 다 있음 → EVIDENCE_SUFFICIENT READY_FOR_BE 유지
execution_node — BE 결과 해석 (Order graph가 아니라 Completion graph 소속)
completion_payload에 execution_result/be_rejection_evidence가 둘 다 있거나 둘 다 없음 (XOR 위반) → COMPLETION_PAYLOAD_INVALID FAILED
be_rejection_evidence 있음 → BE_REVALIDATION_REJECTED BE_REJECTED
execution_result 있음 → ORDER_RESPONSE_VERIFIED REPORT_READY

3 AgentState — 모든 노드가 주고받는 한 덩어리

LangGraph는 state를 노드들이 차례로 변형해서 넘기는 구조. 이 State 모양만 외우면 코드 95% 이해됨. models.py의 AgentState TypedDict.

str | None
HOLD일 때만 의미 있음 (REVIEW / DATA_INSUFFICIENT)
decision_trace

4 6개 Lifecycle 상태

constants.py에 박혀있는 종료/핵심 상태들. 각 노드의 결정이 결국 이 6개 중 하나로 귀결됨.

HOLD

대기 / resume 가능
사람 검토 또는 추가 데이터 필요. hold_reason으로 세분화. 같은 run_id로 resume 가능.

READY_FOR_BE

AI → BE 핸드오프
AI proposal이 BE의 deterministic 재검증으로 넘어가는 상태. 실행 승인 아님.

NO_ORDER

종료
새 주문 없이 종료 (정책 위반 심볼·금액 등). 사용자 설명은 만들 수 있음.

BE_REJECTED

BE 차단
BE 재검증이 차단함. AI가 be_rejection_evidence로 받아 사용자 설명 생성.

FAILED

계약 깨짐 (terminal)
schema mismatch·복구 불가 기술 실패. 같은 run_id 재사용 금지. REPORT_READY로 강제 승격 안 됨.

REPORT_READY

최종 리포트 준비됨
BE 실행 성공 후 사용자용 설명 완성. report.message로 전달.

5 HTTP API — BE가 부르는 3개 엔드포인트

http_api.py가 app.py의 3개 메서드를 그대로 노출. BE가 AI를 사용하는 유일한 통로.

POST /runs/start 새 run 시작 → policy/risk/evaluator 그래프 실행 최초 요청 시
POST /runs/resume HOLD 상태 run에 보완 입력·승인 결과 patch 후 재실행 HOLD 시에만
POST /runs/complete BE의 execution_result 또는 be_rejection_evidence 받아 최종 리포트 생성 READY_FOR_BE 후에만

⚠️ 에러 매핑: unknown run_id → 404, 그 외 ValueError → 400. READY_FOR_BE 이전 complete 호출은 거부됨 (PR #4에서 추가).
