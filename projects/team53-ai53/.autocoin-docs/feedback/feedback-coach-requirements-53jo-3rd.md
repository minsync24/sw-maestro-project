## 1. 핵심 방향: "프롬프트 체이닝"이 아니라 "에이전트 구조"로 보여야 한다

코치가 가장 먼저 지적한 포인트는 현재 오케스트레이터 구조가 **정해진 토폴로지 안에서 일방향으로 흐르는 프롬프트 체이닝처럼 보일 수 있다**는 점이다.

따라서 문서와 발표에서는 아래가 더 분명해야 한다.

- 각 에이전트가 단순히 이전 출력물을 넘겨받는 노드가 아니라, **상황에 따라 리즈닝하고 액션/툴 선택을 수행하는 주체**라는 점
- `Policy / Planning`, `Market`, `Risk`, `Execution / Report` 각각의 **자율성 범위**가 무엇인지
- 어떤 부분은 고정 규칙이고, 어떤 부분은 LLM 판단/툴 선택으로 열려 있는지

즉, 코치 요구사항의 핵심은 "에이전트로 이름만 붙이지 말고, **왜 agentic한지 설명 가능한 구조로 문서화하라**"에 가깝다.

---

## 2. 반드시 보강해야 하는 문서/설계 요구사항

### 2.1 에이전트별 툴 정의를 더 구체화해야 한다

코치는 스펙 문서를 더 구체화하되, 특히 **에이전트별로 어떤 툴을 쓰는지**를 더 디테일하게 적어 달라고 요청했다.

최소한 아래 항목이 문서에 있어야 한다.

- 에이전트별 책임
- 사용할 수 있는 툴 목록
- 어떤 입력을 받아 어떤 출력을 만드는지
- 자율적으로 선택하는 부분과 고정 규칙으로 처리하는 부분

예시 맥락으로는 아래가 언급되었다.

- Market Agent: 정책에 맞춰 RSI, MA 같은 지표 조회/계산 툴을 선택할 수 있어야 함
- Risk Agent: 정책/전략을 바탕으로 리스크 계산을 수행하는 역할이 더 명확해야 함
- Execution / Report Agent: 실행 결과와 리포트 생성 기준이 더 구체적이어야 함

### 2.2 Policy / Planning Agent는 RAG 또는 정책 검색 흐름까지 설명하는 편이 좋다

코치는 Policy / Planning 쪽에서 **문서를 벡터 DB에 넣고**, 사용자 요청과 유사한 투자 정책/기법을 찾아오는 흐름을 제안했다.

문서에는 최소한 아래 정도를 적는 것이 좋다.

- 어떤 정책 문서/전략 문서를 저장하는지
- 사용자 입력과 유사한 정책을 어떻게 찾는지
- 검색된 정책을 Plan 수립에 어떻게 반영하는지

즉, "LLM이 그냥 생각한다"보다 **외부 정책 지식을 검색해 근거로 삼는다**는 구조가 보이면 더 설득력이 높다는 피드백이다.

### 2.3 Evaluator / Reflection 루프를 아키텍처에 추가하는 것이 좋다

코치는 오케스트레이터 안에 **evaluation / self-reflection 루프**가 있으면 더 agentic하게 보인다고 봤다.

핵심 요구사항은 아래다.

- 어떤 에이전트의 출력이 충분히 좋은지 평가하는 단계가 필요함
- 점수 미달이면 이전 단계로 다시 돌아가는 루프를 둘 수 있음
- 특히 Risk 쪽에도 evaluator를 두는 구성이 가능함

문서에 반영할 때는 아래까지 적는 편이 좋다.

- 평가 대상
- 평가 기준
- 점수 방식 예시
- 재시도 조건
- 실패 시 다음 상태

### 2.4 Report 생성 기준은 "언제/무엇을/어떤 단위로"까지 적어야 한다

코치는 Report Agent에 대해 "리포트를 언제 받는지"가 모호하다고 봤다.

따라서 아래를 문서에 명시해야 한다.

- 거래 1건당 리포트인지, 시간 단위 리포트인지
- overnight 실행 결과를 어떤 범위로 묶는지
- 리포트 chunking 단위가 전략 기준인지, 거래 개수 기준인지
- 실패 원인/에러 코드/실행 결과를 어떤 형식으로 남기는지

## 3. 데모 요구사항

코치는 현재 데모가 단순히 "주문 된다 / 취소 된다" 수준으로 보이면 **UI 데모처럼만 보일 수 있다**고 봤다.

그래서 데모는 아래 방향으로 바꾸는 것을 원했다.

- 기능 나열형 데모보다 **정책 기반 시나리오형 데모**로 구성할 것
- 예: 공격적 투자 / 안정적 투자처럼 서로 다른 정책을 입력했을 때
- 각 에이전트가 다른 판단과 다른 워크플로우를 만든다는 점을 보여줄 것

즉, 데모의 목적은 단순 주문 성공이 아니라 **정책에 따라 자율적으로 달라지는 에이전트 워크플로우를 보여주는 것**이다.

---

## 4. 기술 선택 관련 정리

회의에서 확인된 기술 선택 관련 요구사항은 아래와 같다.

- AI 쪽은 가능하면 **LangGraph 사용 유지**
- FE는 Next.js를 꼭 고집할 필요는 없고, **React로 바꿔도 무방**
- 프론트/백엔드 프레임워크 선택보다, 에이전트 구조와 문서 구체화가 더 중요함

즉, 코치가 강하게 본 제약은 FE 프레임워크가 아니라 **agentic 구조가 납득되도록 설계/문서화하는 것**이다.

---

## 5. QA와 구현 방식에 대한 요구사항

코치는 구현 속도 자체보다 **휴먼 QA의 중요성**을 강하게 강조했다.

핵심 요구사항은 아래다.

- AI가 코드를 빠르게 짜는 것만으로 충분하지 않음
- 사람이 직접 에이전트별 입력/출력을 넣어 보며 검증해야 함
- 모듈 단위 QA와 전체 시나리오 QA를 모두 해야 함
- 여러 사람이 같이 붙어서 QA해야 일관성 문제와 숨은 오류를 더 잘 찾을 수 있음

즉, 최종 산출물은 "코드 생성 완료"가 아니라 **사람이 직접 검증한 안정성**까지 포함해야 한다.

---

## 6. 일정성 요구사항

회의에서 드러난 일정 관련 요구사항은 아래와 같다.

- **우선 스펙 문서를 빠르게 완성**할 것
- 스펙은 AI와 티키타카만 하지 말고, 직접 몇 시간 집중해서 읽고 보강할 것
- 데모 제출 일정 전에 문서 정합성을 먼저 올려둘 것
- 중간중간 코드나 문서가 올라오면 코치가 추가 피드백을 줄 수 있음

즉, 구현보다 먼저 **문서를 닫는 작업**이 선행되어야 한다는 뜻으로 읽힌다.

## 7. 우선순위별 실행 항목

### P0. AI/Architecture 문서에서 agentic 근거를 명시한다

- 에이전트별 자율성
- 리즈닝/액션 지점
- 툴 선택 지점
- 고정 규칙 vs LLM 판단 경계

### P1. 에이전트별 툴 계약을 표로 추가한다

- 입력
- 출력
- 사용 툴
- 실패 처리
- 재시도 여부

### P2. Evaluator / Reflection 루프를 구조도와 문서에 반영한다

- 평가 주체
- 평가 점수 기준
- 재시도 흐름
- 실패 시 상태 전이

### P3. 데모 시나리오를 정책 기반으로 재작성한다

- 공격형 / 안정형 등 서로 다른 정책 입력
- 정책에 따라 달라지는 에이전트 행동
- 결과 보고 흐름 포함

### P4. 리포트 및 QA 기준을 명시한다

- 리포트 생성 주기/단위
- 실패 원인 기록 방식
- 사람 중심 QA 시나리오

## 8. 시간 여유가 있으면 반영할 확장 제안

아래 항목은 회의에서 긍정적으로 언급됐지만, 필수보다는 **확장 제안**에 가깝다.

- 과거 리포트/실패 경험을 활용하는 메모리 구조
- 히스토리 DB + 정책 DB를 함께 두는 방식
- 과거 실패 이력까지 반영하는 컨텍스트 엔지니어링

이 부분은 있으면 좋지만, 코치도 회의 맥락상 **핵심은 우선 스펙 구체화와 데모 설득력 확보**라고 본 것으로 읽힌다.

## 확정 구현 기준

이 회의 요구사항을 반영했다고 보려면 최소 아래가 만족되어야 한다.

- 현재 구조가 왜 agentic한지 문서에서 설명 가능하다.
- 에이전트별 툴/입출력/자율성 범위가 문서로 구체화되어 있다.
- 평가 루프 또는 evaluator 개념이 아키텍처와 설명에 반영되어 있다.
- 데모가 단순 기능 시연이 아니라 정책 기반 시나리오를 보여준다.
- 사람 중심 QA 계획이 명시되어 있다.
