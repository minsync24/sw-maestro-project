# Document Review Style Guide

이 프로젝트는 **Coin Agent 문서 레포지토리**이며, 모든 문서는 **Binance Spot Testnet 전용 가상 자금 현물 주문·체결 테스트** 범위를 기준으로 관리한다.
아래 규칙을 기준으로 문서 변경 PR의 리뷰를 수행한다.

---

## Review Comment Style & Language

- 모든 리뷰 코멘트는 **한국어로 작성한다.**
- **무엇이 어긋났는지**, **왜 문제인지**, **어떻게 맞추면 되는지**를 함께 제시한다.
- 사소한 제안에는 `[nit]` 접두어를 붙여 구분한다.
- 이 레포의 문서 리뷰에서는 **범위 통제**, **문서 간 동기화**, **안전 규칙 유지**, **예시 정합성**을 우선 검증한다.

---

## 1) 레포 기본 전제

다음 전제는 모든 문서에서 일관되게 유지되어야 한다.

- 프로젝트는 **Coin Agent** 문서 레포지토리다.
- 거래소 범위는 **Binance Spot Testnet only** 다.
- 실거래 URL, 실거래 API Key / Secret, 선물, 마진, 출금, 레버리지 기능은 문서 범위에서 제외한다.
- 모든 주문은 **가상 자금 기반 현물 주문 테스트**만 다룬다.
- 브라우저는 Binance API를 직접 호출하지 않고, 모든 호출은 **BE를 통해** 수행한다.
- AI는 판단 보조 및 게이트 역할을 수행하며, **실제 실행 권한은 BE에만 있다.**

### 리뷰 체크포인트

| 검사 항목 | 규칙 |
|---|---|
| 범위 일치 | 실거래 기능이나 Production Host가 문서에 섞이면 안 된다. |
| 환경 표기 | REST / WebSocket / WebSocket API 엔드포인트는 Testnet 값만 사용한다. |
| 역할 경계 | AI가 직접 주문 실행 주체처럼 서술되면 안 된다. |
| 실패 기본값 | 정책/리스크 게이트 실패 시 무주문, 판단 보류, 또는 BE 재검증 흐름과 충돌하면 안 된다. |

## 2) 주요 문서와 기대 역할

리뷰 시 문서의 위치와 역할을 먼저 파악하고, 해당 문서가 담당해야 할 정보만 담고 있는지 확인한다.

| 파일명 | 핵심 역할 | 주요 독자 |
|---|---|---|
| `README.md` | 문서 진입점, 읽는 순서, 공통 안전 수칙 | 전원 |
| `PROPOSAL.md` | 기획 배경, 문제 정의, 프로젝트 목적 | 전원 |
| `SPEC.md` | MVP 범위, 기능/비기능 요구사항 | PM, FE, BE, AI |
| `ARCHITECTURE.md` | 전체 구조, 책임 경계, 데이터 흐름 | FE, BE, AI |
| `FE.md` | 화면, 상태, 사용자 흐름, UI 기준 | FE |
| `BE.md` | API, 검증, Testnet 연동 규칙 | BE |
| `AI.md` | Agent 구조, 리스크 게이트, 판단 결과 구조 | AI |
| `DATA.md` | API 계약, 요청/응답, 데이터 모델 | FE, BE, AI |
| `TEST_AND_DEMO.md` | 테스트 체크리스트, 데모 흐름 | 전원 |
| `feedback/*.md` | 기준 문서에 대한 리뷰/비판/개선 제안 기록 | 작성자, 리뷰어 |

### 리뷰 원칙

- 문서는 자신의 역할에 맞는 정보만 담아야 한다.
- 다른 문서가 이미 기준을 정의한 경우, 이를 복제하기보다 **참조 관계를 명확히 유지**해야 한다.
- 같은 개념을 여러 문서에서 설명할 때는 용어와 예시가 동일해야 한다.
- `feedback/*.md` 문서는 기준 문서를 대체하지 않는 **2차 리뷰 문서**로 취급하며, 반드시 대상 문서와 검토 시점을 상단에 명시해야 한다.

### feedback 문서 리뷰 원칙

- feedback 문서는 반드시 현재 저장소의 최신 기준 문서를 직접 참조해야 한다.
- 이미 해소된 이슈와 아직 남아 있는 이슈를 구분해서 써야 한다.
- feedback 문서에 다른 거래소나 이전 프로젝트 문맥이 섞이면 안 된다.
- feedback 문서는 루트 canonical 문서와 분리해 `feedback/` 디렉토리에서 관리한다.
- feedback 문서 상단의 `검토 대상` 목록은 실제 변경 범위 또는 실제 검토 범위와 일치해야 한다.

## 3) 제목, 섹션, 구조 규칙

- 문서는 한국어 중심으로 작성한다.
- 목적이 분명한 도입부 뒤에 구조화된 본문이 이어져야 한다.
- 가능하면 `## 문서 목적`, `## 관련 문서`, `## 확정 구현 기준` 같은 저장소의 반복 패턴을 유지한다.
- 상위 섹션은 일관된 번호 체계를 유지한다.

### 권장 패턴

```markdown
# 문서 제목

## 문서 목적

## 관련 문서

## 1. 주요 섹션

## 2. 세부 규칙

## 확정 구현 기준
```

### 안티패턴

- 문서 목적 없이 본문부터 시작하는 경우
- 같은 문서 안에서 번호 섹션과 비번호 섹션을 뒤섞는 경우
- 제목은 Coin Agent / Testnet 범위인데 본문 예시는 다른 프로젝트 기준인 경우

## 4) 용어, 예시, 엔드포인트 정합성

이 저장소는 도메인 용어와 예시 표기가 엄격한 편이다. 다음 항목은 문서 간 동일해야 한다.

| 항목 | 규칙 |
|---|---|
| REST Base URL | `https://testnet.binance.vision/api` |
| WebSocket Streams | `wss://stream.testnet.binance.vision/ws` |
| WebSocket API | `wss://ws-api.testnet.binance.vision/ws-api/v3` |
| REST 심볼 예시 | `BTCUSDT`, `ETHUSDT`처럼 대문자 |
| Stream 이름 예시 | `btcusdt@ticker`처럼 소문자 |
| 주문 범위 | 현물 주문 테스트만 허용 |

### 리뷰 체크포인트

- 예시 URL이 Production Binance 호스트로 바뀌지 않았는가?
- 심볼 대소문자 규칙이 문서마다 섞이지 않았는가?
- FE / BE / AI 문서에서 같은 요청/응답 필드 이름이 다르게 표기되지 않았는가?
- `NO_ORDER`, `HOLD`, `BE_REJECTED`, `REPORT_READY`, `run_id`, `decision_trace` 같은 핵심 용어가 임의로 바뀌지 않았는가?

## 5) Git / Issue / PR 문서 규칙

이 레포의 협업 규칙은 현재 저장소에 반영된 브랜치/커밋/PR 규칙을 기준으로 본다.

### 브랜치 규칙

- 기본 브랜치는 `main`이다.
- `develop` 브랜치는 사용하지 않는다.
- 이슈 1개당 브랜치 1개를 사용한다.
- 브랜치명은 `{type}/{issue-number}-{description}` 형식을 따른다.
- 브랜치명에는 `#`을 넣지 않는다.

예시:

```text
feat/80-family-detail
fix/82-family-detail-bug
```

### 커밋 규칙

- 커밋 메시지 형식은 `{type}: 작업 내용 (#{issue-number})` 이다.
- 한 커밋에는 한 가지 변경 목적만 담는 것을 권장한다.

### PR 규칙

- PR 제목 형식은 `[#issue-number] 총 작업 내용` 이다.
- PR 본문에는 `Closes #...` 를 포함해야 한다.
- PR 템플릿은 Jira 티켓이 아니라 **GitHub Issue 번호 중심**으로 작성되어야 한다.

### 리뷰 체크포인트

- 템플릿에 Jira, 다른 조직의 브랜치 규칙 같은 외부 프로젝트 흔적이 남아 있지 않은가?
- 브랜치 / 커밋 / PR 규칙이 현재 저장소의 자동화 및 템플릿과 충돌하지 않는가?
- PR 체크리스트가 이 저장소 특성상 중요한 문서 동기화와 안전 점검을 포함하는가?

## 6) 문서 리뷰에서 특히 중요한 실패 유형

다음 항목은 우선적으로 지적한다.

1. **실거래 범위 혼입**
   - Production URL, 실거래 키, 선물/마진/출금 기능이 문서에 등장하는 경우
2. **책임 경계 붕괴**
   - AI가 직접 주문 실행을 담당하는 것처럼 서술되는 경우
3. **문서 간 불일치**
   - SPEC와 DATA, BE와 FE, README와 개별 문서 사이 예시가 다른 경우
4. **검증 불가능한 모호한 표현**
   - “적절히”, “상황에 따라”, “필요 시”처럼 기준 없이 쓰인 문장만 있고 구체 기준이 없는 경우

## 7) 리뷰 코멘트 예시

### 좋은 예

- `실거래 URL인 https://api.binance.com 예시가 들어가 있어 README의 Testnet 전용 규칙과 충돌합니다. 이 문서는 Testnet only 범위이므로 https://testnet.binance.vision/api 로 맞추는 편이 안전합니다.`
- `[nit] PR 템플릿의 Jira 항목은 이 저장소의 GitHub Issue 중심 흐름과 맞지 않습니다. Closes # 형식만 남기면 더 일관됩니다.`

### 피해야 할 예

- `이상합니다.`
- `그냥 바꾸세요.`
- `프로젝트랑 안 맞는 듯?`

---

## 확정 구현 기준

리뷰는 다음 조건을 모두 만족할 때 통과로 본다.

- Coin Agent / Binance Spot Testnet 문맥이 유지된다.
- 실거래 관련 내용이 문서에 포함되지 않는다.
- 문서 간 용어, 예시, 책임 경계가 서로 맞는다.
- Git / Issue / PR 규칙이 현재 저장소의 템플릿과 자동화 규칙과 일치한다.
- 다른 프로젝트의 잔여 문맥이 제거되어 있다.
